难点一：弹窗系统反复出现残留遮罩

问题描述

玩家选择行动（观莲下注、布囊设局）后，选择器弹窗关闭不全——外层的半透明黑色遮罩（含取消按钮）残留在页面上，只显示一个空的选择框和取消按钮。同时结算弹窗（验莲榜）与选择器弹窗叠加，文字完全看不清。

排查过程

逐行追踪 actionPanel._showOverlay() 的 DOM 创建逻辑：

  1. _showOverlay(title, buildFn) 创建了三层 DOM：outer(全屏遮罩) → inner(白色面板) → btnContainer(按钮容器)
  2. 调用 buildFn(btnContainer) 将按钮添加到容器
  3. 但 buildFn 回调中的 overlay.remove() 只删除了 btnContainer（内层容器），外层遮罩节点仍在 DOM 中

解决方案

  1. 在 _showOverlay 中增加 self._activeOverlay 引用，指向外层遮罩
  2. 所有关闭操作统一调用 self._clearOverlay()
  3. 新增 _showOverlay 时先调用 _clearOverlay() 清除上一个弹窗
  4. main.js 显示结算弹窗前主动调用 actionPanel._clearOverlay()
_clearOverlay: function() {
    if (this._activeOverlay) {
        if (this._activeOverlay.parentNode) {
            this._activeOverlay.parentNode.removeChild(this._activeOverlay)
        }
        this._activeOverlay = null
    }
}

难点二：结算弹窗内容渲染在白色框外

问题描述

验莲榜弹出后，标题在白色消息框内，但下面的排名、铜钱文字全部渲染在半透明黑底上，因对比度不足而看不清。用户反复反馈"下面的文字看不清楚"。

排查过程

检查 DOM 追加逻辑：_createOverlay() 返回的是外层遮罩 overlay（全屏半透明黑底），而 showRoundSettle() 将排名、铜钱等子元素通过 overlay.appendChild() 追加到了外层，而非内层白色框。

解决方案

重构 _createOverlay 为 _makeOverlay，返回结构化对象：

return { outer: outerDiv, box: innerBox }

所有内容统一追加到 box（白色内框），outer 仅最后一步 appendChild 到容器。同时提高所有文字的字号（20-24px）、颜色深度（#1a1010），龟名使用专门映射的深色替代浅色原色。

难点三：AI系统间歇性冻结

问题描述

玩家完成行动后，后续AI有时会停止响应——轮到AI回合但没有任何动作，页面静止不动。概率性出现。

排查过程

分析 AI 决策时序：

  1. AI 的 update() 设置 state.ai._thinking = true，启动 1500ms 定时器
  2. 定时器回调执行 _decide() 后置 _thinking = false
  3. 但 _decide() 执行期间，回合阶段可能切换为 settle
  4. 定时器触发时回合已结束，_decide() 在不合法的阶段执行操作
  5. 异常导致 _thinking = false 永远不执行
  6. 此后每帧被 if (state.ai._thinking) return 拦截，AI 永久停止

解决方案（三层防护）

第一层：定时器回调开头检查阶段合法性

setTimeout(function() {
    if (state.round.phase !== 'player_turn') {
        state.ai._thinking = false; return;
    }
    // ...
}, 1500)

第二层：_decide() 加 try-catch 兜底

try { self._decide(state, currentIdx) } catch (e) {}
state.ai._thinking = false  // 无论如何都清除

第三层：主循环在结算和结束时强制复位

store.state.ai._thinking = false

难点四：多局游戏间的状态污染

问题描述

完成一局后，再开一局，上一局的赛况录内容依然存在，新旧日志混杂显示。

排查过程

  • resetGame() 只移除了 DOM 面板
  • startGame() 的 resetState 只重置数据，不清理订阅者
  • historyLog._lastRendered 计数器未清零
  • logSystem.init() 的 store.subscribe() 订阅者累积未清除

解决方案

  1. startGame() 开头增加 store.clearAll() 清除旧 handler 和 subscriber
  2. historyLog.mount() 检测已有面板并移除，重置 _lastRendered = 0

难点五：8龟叠罗汉 + 逆行龟推动的逻辑实现

问题描述

从5只骆驼扩展为6只普通龟 + 2只逆行龟。逆行龟的"相遇推动"机制需要在不破坏叠罗汉语义的前提下实现。

设计决策

  • 逆行龟与普通龟使用相同的格子数组 state.race.cells[cellIdx]
  • 移动方向由参数 forward 控制
  • 逆行龟移动后调用 _pushForward()：扫描目标格，将非逆行龟类向前推1格
  • 推动时从栈顶向下逐个处理，避免索引偏移
_pushForward: function(state, cell) {
    // 先收集目标格中所有普通龟
    for (var i = 0; i < stack.length; i++) {
        if (!isSpecial(tid)) toPush.push({ id: tid, idx: i })
    }
    // 从栈顶往下逐个推进1格
    for (var p = toPush.length - 1; p >= 0; p--) {
        var pushed = cells[cell].splice(idx, 1)
        cells[nextCell].push(pushed[0])
    }
}

难点六：签筒余签面板与回合同步

新增"签筒余签"面板在每轮显示哪些龟还未被抽出。signsRemaining 中每支签标记 isSpecial 字段，逆行签用黑白渐变标签，普通龟用各自颜色标签。面板每次重绘按颜色去重显示。

难点七：Canvas尺寸与赛道文字可读性

赛道从16格扩展到25格后，Canvas 960×640 空间不足。解决：Canvas 扩大至 1100×700，所有面板同步扩展，赛道标签字号从 10px 提升至 14px bold，终点用红色突出。

经验总结

  1. DOM弹窗管理需要统一的引用追踪：不能依赖闭包参数判断当前DOM节点。
  2. 异步操作与状态机的边界需严格检查：定时器回调必须验证阶段合法性。
  3. 多局游戏间的状态清理必须彻底：数据、订阅者、DOM残留、计数器都要清理。
  4. Canvas文字可读性依赖足够的物理像素：25格赛道需至少1100px宽度。
  5. 测试先行可大幅减少回归：42项测试每次修改后运行。

# 莲池竞渡 — 设计难点与解决方案

## 难点一：弹窗系统反复出现残留遮罩

### 问题描述
玩家选择行动（观莲下注、布囊设局）后，选择器弹窗关闭不全——外层的半透明黑色遮罩（含取消按钮）残留在页面上，只显示一个空的选择框和取消按钮。同时结算弹窗（验莲榜）与选择器弹窗叠加，文字完全看不清。

### 排查过程
逐行追踪 `actionPanel._showOverlay()` 的 DOM 创建逻辑：
1. `_showOverlay(title, buildFn)` 创建了三层 DOM：`outer(全屏遮罩) → inner(白色面板) → btnContainer(按钮容器)`
2. 调用 `buildFn(btnContainer)` 将按钮添加到容器
3. 但 buildFn 回调中的 `overlay.remove()` 只删除了 btnContainer（内层容器），外层遮罩节点仍在 DOM 中

### 解决方案
1. 在 `_showOverlay` 中增加 `self._activeOverlay` 引用，指向外层遮罩
2. 所有关闭操作统一调用 `self._clearOverlay()`，确保完整移除整棵 DOM 树
3. 新增 `_showOverlay` 时先调用 `_clearOverlay()` 清除上一个弹窗
4. `main.js` 显示结算弹窗前主动调用 `actionPanel._clearOverlay()`

```javascript
_clearOverlay: function() {
    if (this._activeOverlay) {
        if (this._activeOverlay.parentNode) {
            this._activeOverlay.parentNode.removeChild(this._activeOverlay)
        }
        this._activeOverlay = null
    }
}
```

## 难点二：结算弹窗内容渲染在白色框外

### 问题描述
验莲榜弹出后，标题"验莲榜"在白色消息框内，但下面的排名、铜钱文字全部渲染在半透明黑底上，文字因对比度不足而看不清。用户反复反馈"下面的文字看不清楚"。

### 排查过程
检查 DOM 追加逻辑：`_createOverlay()` 返回的是外层遮罩 `overlay`（全屏半透明黑底），而 `showRoundSettle()` 将排名、铜钱等子元素通过 `overlay.appendChild()` 追加到了外层，而非内层白色框。

### 解决方案
重构 `_createOverlay` 为 `_makeOverlay`，返回结构化对象：
```javascript
return { outer: outerDiv, box: innerBox }
```
所有内容（排名、铜钱、按钮）统一追加到 `box`（白色内框），`outer` 仅最后一步 `appendChild` 到容器。同时提高所有文字的字号（20-24px）、颜色深度（`#1a1010`），龟名使用专门映射的深色替代浅色原色。

## 难点三：AI系统间歇性冻结

### 问题描述
玩家完成行动后，后续AI有时会停止响应——轮到AI回合但没有任何动作，页面静止不动。不是每次都发生，概率性出现。

### 排查过程
分析 AI 决策时序：
1. AI 的 `update()` 在轮到 AI 时设置 `state.ai._thinking = true`，并启动 1500ms 定时器
2. 定时器回调执行 `_decide()` 后置 `_thinking = false`
3. 但 `_decide()` 执行期间，其他玩家的 `NEXT_PLAYER` dispatch 可能导致回合阶段切换为 `settle`
4. 定时器触发时回合已结束，`_decide()` 在不合法的阶段执行操作
5. 异常导致 `_thinking = false` 永远不执行
6. 此后每帧 `update()` 被 `if (state.ai._thinking) return` 拦截，AI 永久停止

### 解决方案（三层防护）

**第一层**：定时器回调开头检查阶段合法性
```javascript
setTimeout(function() {
    if (state.round.phase !== 'player_turn') {
        state.ai._thinking = false
        return
    }
    // ...
}, 1500)
```

**第二层**：`_decide()` 加 try-catch 兜底
```javascript
try {
    self._decide(state, currentIdx)
} catch (e) {
    // 吞掉异常，确保后续代码执行
}
state.ai._thinking = false  // 无论如何都清除标记
```

**第三层**：主循环在回合结算和游戏结束时强制复位
```javascript
// main.js — 结算前
store.state.ai._thinking = false
```

## 难点四：多局游戏间的状态污染

### 问题描述
完成一局单人游戏后，再开一局多人热座，上一局的赛况录（历史记录面板）内容依然存在，新旧日志混杂显示。

### 排查过程
1. `resetGame()` 只移除了 DOM 面板并重新显示了开始页
2. `startGame()` 调用 `store.resetState(seed)` 重置了数据状态
3. 但 `historyLog._lastRendered` 计数器未清零，DOM 列表元素也未清空
4. 更严重的是：`logSystem.init()` 注册的 `store.subscribe()` 订阅者累积未清除，旧订阅者继续向新游戏写入日志

### 解决方案
1. `startGame()` 开头增加 `store.clearAll()` 彻底清除旧的 handler 和 subscriber
2. `historyLog.mount()` 时检测已有面板并移除，重置 `_lastRendered = 0`

## 难点五：8龟叠罗汉 + 逆行龟推动的逻辑实现

### 问题描述
原版5只骆驼，扩展为6只普通龟 + 2只逆行龟（从终点反向移动）。逆行龟的"相遇推动"机制需要在不破坏叠罗汉语义的前提下实现。

### 设计决策
- 逆行龟与普通龟使用相同的格子数组 `state.race.cells[cellIdx] = [turtleId, ...]`
- 移动方向由参数 `forward` 控制
- 逆行龟移动后调用 `_pushForward()`：扫描目标格，将非逆行龟类向前推1格
- 推动时从栈顶向下逐个处理，避免索引偏移

### 关键代码
```javascript
_pushForward: function(state, cell, excludeIds) {
    // 先收集目标格中所有普通龟
    for (var i = 0; i < stack.length; i++) {
        if (!isSpecial(tid)) toPush.push({ id: tid, idx: i })
    }
    // 从栈顶往下逐个推进1格
    for (var p = toPush.length - 1; p >= 0; p--) {
        var pushed = cells[cell].splice(idx, 1)
        cells[nextCell].push(pushed[0])
    }
}
```

## 难点六：签筒余签面板与回合同步

### 问题描述
新增"签筒余签"面板（右下角）需要在每轮显示哪些龟还未被抽出。但签筒状态在 `state.round.signsRemaining` 中，且包含特殊逆行签。

### 解决方案
1. `signsRemaining` 中每支签标记 `isSpecial` 字段
2. 逆行签用黑白渐变标签识别，普通龟用各自颜色标签
3. 面板在 `unactedPanel.render()` 中每次重绘，按颜色去重显示

## 难点七：Canvas尺寸与赛道文字可读性

### 问题描述
赛道从16格扩展到25格后，Canvas 960×640 空间不足。起点"起莲台"和终点"摘莲台"文字小到无法辨认。

### 解决方案
1. Canvas 扩大至 1100×700（约 15% 增幅）
2. 所有固定尺寸面板同步扩展（playerBar、actionPanel、settlementOverlay、startScreen）
3. 赛道标签字号从 10-11px 提升至 14px bold
4. 终点文字改用红色（`#c04040`）以突出显示
5. 轨道padding从60px减至50px以腾出更多赛道空间

## 难点八：游戏设计从骆驼到灵龟的主题转换

### 挑战
将埃及沙漠骆驼赛跑完整转换为中国古代江南水乡的灵龟赛跑，需要保持所有机制不变的同时更换全部视觉元素和文案。

### 映射关系

| 原版 | 中国古风版 |
|------|-----------|
| 骆驼 ×5 | 灵龟 ×6（青玄/赤霄/黄枢/白泽/墨渊/紫电）取五行五色 |
| 沙漠赛道 | 莲花池塘16段（→25段） |
| 金字塔骰塔 | 竹制签筒 |
| 绿洲/海市蜃楼 | 浮萍(+1)/漩涡(-1) |
| 金币 | 铜钱（外圆内方） |
| 分段赛/全程竞猜 | 观莲下注/一卦定局 |
| 总冠军/吊车尾 | 鳌头/殿后 |
| 回合结算 | 验莲榜 |
| 沙漠图板 | 锦囊 |

所有文案统一使用古风术语，视觉配色从沙金色调整为宣纸米白+墨色+朱砂红。

## 难点九：视觉风格统一换代

### 问题描述
初始版本使用纯色背景、系统默认字体、粗边框和大阴影，视觉效果呆板。

### 改进方案

| 方面 | 旧版 | 新版 |
|------|------|------|
| 字体 | Inter/system | KaiTi/STKaiti（楷体）+ FangSong（仿宋） |
| 背景 | `#f5f0e8` 纯色 | 五阶渐变 `#e8dfd2 → #f7f3eb → #dfd4c0` |
| 面板边框 | 2px `#8b7355` | 1px `rgba(139,115,85,0.15-0.3)` |
| 面板阴影 | `0 2px 8px rgba(...)` | `0 1px 2px rgba(..., 0.06)` |
| 按钮背景 | 纯色 `#c04040` | 纵向渐变 `#c84848 → #a83838` |
| 按钮hover | 仅变色 scale(1.05) | translateY(-1px) scale(1.04) brightness(1.08) |
| 过渡曲线 | linear | cubic-bezier(.34,1.56,.64,1)（微弹性） |

## 经验总结

1. **DOM弹窗管理需要统一的引用追踪**：不能依赖闭包参数来判断当前操作的是哪个DOM节点，需要显式的 `_activeOverlay` 引用。

2. **异步操作（setTimeout）与状态机的边界需严格检查**：定时器回调必须验证当前阶段是否仍然合法。

3. **多局游戏间的状态清理必须彻底**：不仅清理数据状态，还要清理订阅者、DOM残留、渲染计数器。

4. **Canvas文字可读性依赖足够大的物理像素**：25格赛道在960px宽度下每格仅38px，1100px下每格42px，文字从不可读到勉强可读。

5. **测试先行可大幅减少回归**：42项测试在每次改动后运行，确保核心逻辑不受UI层修改影响。

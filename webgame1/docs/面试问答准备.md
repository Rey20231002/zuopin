# 莲池竞渡 — 面试问答准备

---

## Q1：你是怎么利用 AI 完成这个项目的？你在其中起到了什么作用？

**核心观点**：AI 是执行者，我是决策者。我负责"做什么"和"为什么"，AI 负责"怎么做"。

**具体分工**：

| 我主导 | AI 执行 |
|--------|---------|
| 产品方向决策（主题、玩法、赛道格数） | 代码编写（21个JS文件） |
| 玩法平衡设计（4种行动收益、AI性格参数） | 测试用例生成 |
| 视觉风格定调（色彩、字体、动效曲线） | DOM/CSS 具体实现 |
| Bug定位与修复策略 | 按我的指令修改代码 |
| 质量验收标准 | 运行测试、浏览器验证 |

**工作流举例**：
1. 我提出需求："弹窗关闭后有残留遮罩"
2. 我指导排查方向："检查 _showOverlay 的 DOM 创建逻辑"
3. AI 读取代码、分析问题
4. 我确认根因（btnContainer vs outer overlay 的引用错误）
5. 我给出修复方案："增加 _activeOverlay 引用，统一用 _clearOverlay()"
6. AI 执行代码修改
7. 我验收效果

**关键能力体现**：不是"让AI写了一个游戏"，而是"我借助AI作为加速器，完成了从设计到交付的全流程"。

## Q2：弹窗残留这个bug你是怎么解决的？

**问题**：选完行动后，选择器弹窗只关闭了内部按钮容器，外层的半透明遮罩和取消按钮残留在页面上。

**排查过程**：
1. 首先怀疑是 CSS z-index 或 DOM 未移除
2. 让 AI 打印 `_showOverlay` 的完整 DOM 创建逻辑
3. 发现第 337 行 `buildFn(btnContainer)` — 传给回调的是按钮容器，不是外层遮罩
4. 所有按钮点击时调用 `overlay.remove()`，但 `overlay` 变量指向的是 `btnContainer`

**修复**：
1. 增加 `this._activeOverlay` 显式引用外层遮罩
2. 新增 `_clearOverlay()` 统一关闭方法
3. 打开新弹窗前先清旧弹窗
4. 结算弹窗显示前主动清理

**教训**：DOM 弹窗管理不能依赖闭包参数隐式传递引用，需要显式的状态追踪。

## Q3：AI 间歇性冻结是怎么排查出来的？

**问题**：概率性出现——AI 轮到回合但页面静止不动。

**排查思路**：这类"间歇性"bug 通常和异步时序有关。AI 用 `setTimeout(fn, 1500)` 做决策延迟，我怀疑是定时器触发时游戏状态已经变了。

**验证过程**：
1. 让 AI 画出完整的时序图：玩家行动 → NEXT_PLAYER → AI update → setTimeout → _decide
2. 发现关键路径：setTimeout 触发时，回合可能已经结算（phase 变为 settle）
3. `_decide()` 在 settle 阶段执行操作导致异常，`_thinking = false` 永远不执行
4. 此后每帧被 `if (state.ai._thinking) return` 拦截

**修复（三层防护）**：
- 定时器回调开头检查 `phase !== 'player_turn'` 则放弃
- try-catch 兜底确保 `_thinking` 必定复位
- 回合结算时强制 `state.ai._thinking = false`

**教训**：异步操作与状态机的边界必须严格校验，不能假设调用时的状态在回调时仍然有效。

## Q4：结算弹窗文字叠加是怎么解决的？

**问题**：验莲榜弹窗中，排名和铜钱文字渲染在半透明黑底上（而非白色消息框内），完全看不清。

**排查**：发现 `_createOverlay()` 返回的是外层遮罩 DOM，而 `showRoundSettle()` 把所有子元素 `appendChild` 到了外层。内层白色框和外层遮罩是父子关系，文字被追加到了错误层级。

**修复**：重构 `_createOverlay` 为 `_makeOverlay`，返回 `{ outer, box }` 结构对象。所有内容统一追加到 `box`（白色内框），彻底杜绝层级错误。

**教训**：UI 组件的 API 设计要明确区分"容器"和"内容区"，不能靠约定来保证正确性。

## Q5：这个项目的架构设计有什么特点？

**核心约束**：纯前端、零依赖、`file:///` 直开。这意味着不能用 ES modules、不能用 npm、不能用构建工具。

**架构选择**：

1. **自研状态管理**（100 行）：而非引入 Redux/Vuex
   - 事件派发 + 切片订阅 + 确定性 PRNG
   - 支持完整游戏回放（seed + history 即可重现）

2. **单向数据流**：`dispatch → handler → state → subscriber → UI`
   - UI 层只能 `dispatch` 和 `subscribeTo`，禁止直接写状态
   - 系统层只能修改自己声明的状态切片

3. **状态纯度**：`store.state` 只能包含可序列化数据（无 DOM 引用、无类实例）

4. **Config vs State 分离**：静态数值在 config，动态值在 state

**取舍**：用 IIFE + window 全局变量替代 ES modules — 这是 `file:///` 兼容的代价。

## Q6：这个项目有什么缺点和可以改进的地方？

**坦诚回答 + 改进思路**：

1. **没有后端，无法在线多人**
   - 改进：加 WebSocket 服务端 + 房间匹配
   - 当前热座模式只适合当面玩

2. **AI 决策比较简单（启发式规则）**
   - 改进：可以用蒙特卡洛树搜索（MCTS）模拟推演未来签序
   - 但计算量在浏览器端要控制

3. **Canvas 渲染没有做视口适配**
   - 1100×700 固定尺寸，小屏幕上需要缩放
   - 改进：根据 `window.innerWidth` 动态调整 Canvas 大小

4. **没有音效和音乐**
   - 古风背景音乐 + 抽签/下注/结算音效会大幅提升沉浸感

5. **测试覆盖不够全面**
   - 42 项测试覆盖了核心逻辑，但 UI 交互、AI决策正确性没测
   - 改进：加 E2E 测试（Playwright）

6. **代码组织可以更好**
   - 21 个文件全在两层目录下，系统再复杂就需要分层
   - 改进：引入子系统目录（如 `systems/race/`, `systems/bet/`）

## Q8：你从中学到了什么？

1. **Vibe Coding 的核心不是"让AI写代码"，而是"我定义问题边界，AI在边界内执行"**。需求越清晰、约束越具体，AI 的产出质量越高。

2. **测试是 AI 辅助开发的生命线**。42 项测试在每次改动后运行，多次拦截了 AI 引入的回归bug。

3. **状态机 + 异步操作的边界校验是前端复杂交互的常见陷阱**。AI 冻结的 bug 本质就是定时器回调没有验证当前状态。

4. **DOM 层级管理需要显式的引用追踪**，不能依赖闭包隐式绑定。弹窗残留 bug 三次才彻底修好。

5. **Game Design 和 Game Development 是两种不同的思维方式**：前者要回答"什么是有趣的"，后者要回答"怎么实现它"。这个项目逼我同时切换两种视角。
